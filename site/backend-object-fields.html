Backend Object Fields

A simple cheat sheet for common backend objects, their fields, and the logic they usually carry.

LogEntry

  • timestamp
  • message
  • serverId

Logic: Used as the base object in a Min-Heap for K-Way Merging. Sort by timestamp ascending.

LoginTracker

  • userId
  • failed timestamps
  • maybe lock-related state if needed

Logic: isLocked() returns whether the current time is before lockUntil. recordFailure(now) adds the new timestamp and can update the lock state.

SessionRecord

  • sessionId
  • userId
  • expiryTime
  • invalidated / active

Logic: isValid() checks active state and expiry. refresh(duration) extends the session.

TokenRecord

  • token
  • userId
  • expiryTime
  • revoked

Logic: Used for auth validation, expiry checks, and revocation handling.

RateLimitRecord

  • userId or clientId
  • request timestamps
  • window size
  • limit

Logic: Slide the window by removing old timestamps, then check whether the request can still be allowed.

RequestRecord

  • requestId
  • payload hash
  • createdAt
  • status

Logic: Track whether a request is pending, completed, or already processed to support deduplication and idempotency.

CacheNode

  • key
  • value
  • prev
  • next

Logic: detach() removes the node from the list. promote() moves it to the head for LRU-style usage.

HeapEntry

  • timestamp or priority
  • source / server index
  • item index
  • object reference if needed

Logic: After polling the top item, use the source and item index to fetch the next candidate from that same stream.

RetryTask

  • taskId
  • payload
  • retryCount
  • nextRunTime

Logic: Use exponential backoff to schedule the next retry and stop after a max retry limit.

DistributedLock

  • lockKey
  • ownerId
  • ttl

Logic: Acquire the lock only if it does not exist, and release it safely only if the caller still owns it.
