Interview problem: design URL shortener. Key components: ID generation, storage, redirect. For ID generation, use base62 encoding of an auto-incremented ID — gives short, unique, non-guessable URLs. Storage: a simple key-value store works (short code → original URL). Redis for fast reads with TTL, backed by a database for persistence. Redirect: 301 (permanent) vs 302 (temporary) — use 301 if you want browsers to cache, 302 if you want to track every click server-side. Scale consideration: reads are much more frequent than writes, so optimize for read path. Add a cache layer in front of the database. Lesson: URL shortener is a good template for many read-heavy key-value lookup systems.
