Interview problem: design a rate limiter. I initially overcomplicated it by thinking about distributed state too early. The interviewer wanted to see the basic algorithm first. The sliding window approach works well: store a timestamp log of requests per user, and on each new request, remove timestamps older than the window, then check if the count is within the limit. For distributed systems you would use Redis with atomic operations. Lesson: in system design interviews, start simple and layer complexity only when the interviewer asks. Show that you understand the core algorithm before jumping to distributed edge cases. Always ask about scale requirements before designing anything.
